# OverVK

SOCKS5-туннель поверх сообщений ВКонтакте. Один бинарник `overvk` запускается в режиме клиента или сервера по YAML-конфигу.

## Сборка

Требуется Go 1.22 или новее.

```bash
go build -o bin/overvk ./cmd/overvk
```

Проверка:

```bash
go test ./...
go vet ./...
```

## Запуск

```bash
./bin/overvk path/to/config.yaml
```

Роль процесса выбирается полем `mode`:

```yaml
mode: client
```

или:

```yaml
mode: server
```

## Конфиг клиента

```yaml
mode: client
token: <client-community-token>
key: "shared-secret"
group: 123456789
handshake_phrase: "/overvk-peer"
verbose: false

engine:
  socks_host: 127.0.0.1
  socks_port: 8888
  sender_workers: 16
  sender_queue_size: 1000
  max_chunk_size: 1048576
  chunk_timeout: 150ms
  text_message_threshold: 6144
  upload_url_cache_ttl: 5m
  max_packet_buffer_size: 100
  packet_buffer_timeout: 30s
  http_timeout: 90s
  ack_wait: 60s
  long_poll_wait: 25
  long_poll_delay: 5s
  bootstrap_peer_scan_limit: 500
  bootstrap_collect_timeout: 3s
  enable_session_statistics: true
  api_version: "5.131"
```

После запуска клиент получает список своих VK-бесед, рассылает в них служебный сигнал `ready`, собирает ответы сервера `ready_ack` из общих бесед и только потом начинает слушать SOCKS5:

```text
127.0.0.1:8888
```

## Конфиг сервера

```yaml
mode: server
token: <server-community-token>
key: "shared-secret"
group: 987654321
handshake_phrase: "/overvk-peer"

  sender_queue_size: 1000
  max_chunk_size: 1048576
  chunk_timeout: 150ms
  text_message_threshold: 6144
  upload_url_cache_ttl: 5m
  max_packet_buffer_size: 100
  packet_buffer_timeout: 30s
  http_timeout: 90s
  long_poll_wait: 25
  long_poll_delay: 5s
  bootstrap_peer_scan_limit: 500
  bootstrap_collect_timeout: 3s
  enable_session_statistics: true
  api_version: "5.131"
```

Все поля `engine` опциональны. Если поле не указано, используется значение по умолчанию.

## Шифрование

Если в конфиге задан `key`, все VK-сообщения шифруются AES-256-GCM. Шифруется всё содержимое сообщения целиком, включая служебные заголовки протокола (`To`, `Type`, `SessionID`, `MessageID`), что делает трафик неотличимым от случайных данных для стороннего наблюдателя.

Ключ должен совпадать на клиенте и сервере. Из переданной строки выводится 256-битный ключ через SHA-256.

Генерация секрета:

```bash
openssl rand -base64 32
```

Если `key` не задан, шифрование отключено и протокол работает в открытом виде (обратная совместимость).

Документы (крупные чанки) шифруются отдельно: заголовочный блок сообщения и содержимое файла шифруются независимо.

## Bootstrap бесед

`peer_id` не задаётся в конфиге. Клиент получает беседы через VK API, дополнительно батчами сканирует локальные chat peer ID от `2000000001` до лимита `bootstrap_peer_scan_limit` и отправляет туда протокольное приветствие. Сервер отвечает во всех общих беседах, где увидел приветствие.

После первого совпадения обе стороны ещё ждут `bootstrap_collect_timeout`, собирают остальные общие беседы и затем используют весь найденный список циклически для дальнейшего обмена. Каждая следующая отправка сообщения выбирает следующий найденный чат.

`bootstrap_peer_scan_limit` задаёт верхнюю границу локальных номеров бесед для скана. Например, `500` означает проверку диапазона `2000000001` ... `2000000500`. Если общих бесед много, но OverVK находит не все, увеличьте это значение.

`bootstrap_collect_timeout` задаёт, сколько ждать после первого найденного общего чата, чтобы собрать остальные ответы. Если чатов много или VK отвечает неравномерно, увеличьте до `5s` или `10s`.

`handshake_phrase` используется только как маркер служебных сообщений `ready`/`ready_ack`; значение должно совпадать в клиентском и серверном конфиге. Если поле не указано, используется `/overvk-peer`.

Важно: `peer_id` одной физической беседы может отличаться для разных сообществ. Например, клиент может видеть общий чат как `2000000002`, а сервер как `2000000001`. Это нормально: OverVK использует те `peer_id`, которые VK прислал конкретному сообществу.

## Подготовка VK

Нужно два сообщества VK: одно для клиента, второе для сервера.

1. В каждом сообществе откройте `Управление` -> `Работа с API`.
2. Включите Long Poll API.
3. В типах событий включите `Входящее сообщение`.
4. Создайте ключ доступа с правами `сообщения` и `управление`.
5. В `Управление` -> `Сообщения` включите сообщения сообщества.
6. В настройках бота включите возможность добавлять сообщество в беседы.
7. Создайте один или несколько общих чатов и добавьте туда оба сообщества. Несколько чатов обычно заметно быстрее, потому что отправка сообщений распределяется по разным беседам.
8. Запустите клиентский и серверный процессы.
9. Клиент отправит `ready` во все свои беседы, сервер ответит `ready_ack` во всех найденных общих беседах; после этого оба процесса выведут найденные peer ID и начнут циклически использовать эти беседы.

Не берите номер беседы из адресной строки VK. Для сообщества надёжный источник — `peer_id` из события `message_new`, полученного именно этим сообществом через Group Long Poll.

## Systemd для сервера

Пример установки:

```bash
sudo mkdir -p /var/overvk-server
sudo cp bin/overvk /var/overvk-server/
sudo cp server.config.yaml /var/overvk-server/config.yaml
sudo cp overvk.service /etc/systemd/system/overvk.service
sudo systemctl daemon-reload
sudo systemctl enable --now overvk.service
```

Проверить статус:

```bash
sudo systemctl status overvk.service
```

Смотреть логи:

```bash
journalctl -u overvk.service -f
```

## Как это работает

Клиент принимает локальные SOCKS5-подключения и отправляет `CONNECT` серверу через VK. После подтверждения трафик режется на чанки:

- небольшие чанки кодируются в Base64 и отправляются текстовыми сообщениями;
- крупные чанки отправляются как VK-документы;
- сообщения получают `SessionID`, `MessageID`, направление и тип;
- при включённом шифровании всё содержимое каждого сообщения шифруется AES-256-GCM;
- принимающая сторона дешифрует, восстанавливает порядок пакетов и пишет данные в TCP-сокет.

Сервер слушает Long Poll своего сообщества, получает команды клиента, открывает TCP-соединение к целевому хосту и возвращает ответ клиенту тем же протоколом.

## Производительность

VK может включать flood control при слишком частых `messages.send`, особенно если много сообщений идёт в одну беседу. Для скорости лучше использовать несколько общих бесед: OverVK найдёт их на bootstrap и будет ротировать отправку по кругу.

Для проверки лучше начинать не с обычного браузера, а с точечного запроса:

```bash
curl --socks5-hostname 127.0.0.1:8888 http://ifconfig.me
```

Браузер может параллельно открыть десятки соединений к служебным доменам, что быстро упирается в лимиты VK.
